Since the error is identical even after setting the environment variable, something else is likely throwing during app startup before EF even gets a working context — and that real exception is getting swallowed into this generic message. The way to unmask it is --verbose.

Run this from the Web folder:

bash
dotnet ef database update -c catalogcontext -p ../Infrastructure/Infrastructure.csproj -s Web.csproj --verbose
Scroll up through the (long) output looking for the first real exception — it'll usually be well above the final "Unable to resolve service for type IMigrator" message. That earlier exception is the actual root cause; the IMigrator message is just EF's generic fallback when it couldn't finish building your app's service provider at all.

A likely candidate, looking at Program.cs: right before builder.Build(), there's this:

    BaseAddress = new Uri(baseUrlConfig!.WebBase)

GetRequiredSection throws if the BaseUrlConfiguration section is missing from config, and new Uri(null) throws if WebBase isn't set. If your appsettings.json (or appsettings.Development.json) in the Web folder is missing or incomplete, this crashes before the app finishes building — which would produce exactly the symptom you're seeing, unrelated to the DbContext at all.

Also worth double-checking: did you set ASPNETCORE_ENVIRONMENT=Development in the same terminal window/session you're running the dotnet ef command from? Environment variables set with set/$env:/export only apply to that specific shell instance — if you opened a new terminal or ran it via an IDE's integrated terminal that reset, it won't have carried over. Run echo $env:ASPNETCORE_ENVIRONMENT (PowerShell) or echo %ASPNETCORE_ENVIRONMENT% (cmd) right before the ef command to confirm it's actually set to Development in that session.
